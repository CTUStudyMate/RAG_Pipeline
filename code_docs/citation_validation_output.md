Each answer generated by the LLM (when RAG is used) consists of multiple segments. Each segment is a claim derived directly from the data source or synthesized from ideas within the data source.

The LLM returns a list of segments, which are then validated for compliance. Segments that fail validation are discarded, including:
- Segments whose evidence is invalid source text (does not exist in the data source)
- Segments whose evidence is an invalid image (image ID does not exist)

This filtering ensures that information fabricated by the LLM — including fabricated citations — is removed.

After citation validation, the answer takes the following structure:

[
  {
    "role": "paragraph | sentence | bullet | bullet_intro",
    "segment": "the LLM's claim content",
    "type": "cited | inferred | intro",

    "citations": [
      {
        "type": "source_text",
        "content": "...",
        "processed_texts": [
          (doc_id, "matched text snippet from source")
        ]
      },
      {
        "type": "img",
        "img_id": "image_0_1",
        "processed_info": [
          (doc_id, "image_0_1")
        ]
      }
    ],

    "processed_cite_obj": {
      "texts": {
        "doc_1": ["text snippet 1", "text snippet 2"]
      },
      "images": {
        "doc_2": ["img_1", "img_2"]
      }
    }
  },
  {
    ....
  }
]

After processing, segments are assigned one of the following types:

🟢 `"type": "cited"`
- The segment has valid citations
- At least one text or image was matched within the docs
- Has a `processed_cite_obj`

🟡 `"type": "inferred"`
- The segment has no citations
- Cannot be verified against any source

🔵 `"type": "intro"`
- Reserved for `bullet_intro` segments with no citations
- Typically used for headings or section openers

🔴 `"type": "abstained"`
- The segment conveys that the system declined to answer the question — this may be due to insufficient information, a policy violation, or a system error during processing

💬 `"type": "conversational"`
- Used when the LLM's response addresses user messages that do not require RAG, such as greetings, expressions of thanks, encouragement, etc.

---

The `processed_cite_obj` field is designed to make UI rendering straightforward. For text citations, each `chunk_id` is treated as a distinct cite mark. Therefore, if a segment's citations include source text from two different chunks, that segment may be marked as `[1][2]`. Each image citation is assigned its own separate mark.
